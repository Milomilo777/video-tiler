## What it is
Desktop app that downloads one live video stream once and tiles it as an N×N grid of identical, synchronized windows (optionally spread across multiple monitors) — built for playing the Supreme Master TV live stream on a wall of screens.

A Tkinter GUI drives a single `yt-dlp` download piped through `ffmpeg`/`ffplay`, fanned out to N² borderless always-on-top player windows (one process per monitor when multi-monitor is on). Ships kiosk-grade reliability: offline connectivity probing with a looping local fallback video, auto-restart/self-heal per dead window, sleep prevention, exact per-monitor DPI-aware fullscreen, and adaptive quality stepping on weak CPUs.

## Run it
```
pip install -r requirements.txt
python src/video-tiler.py        # or double-click run.bat on Windows

# Windows standalone .exe:
compile_windows.bat              # -> dist\video-tiler.exe (PyInstaller)

# Tests:
run_tests.bat                    # or: python tests/run_tests.py
```

## Architecture
Two-class core in `src/video-tiler.py`: `Player` (the playback engine — spawns yt-dlp/ffmpeg/ffplay, fans one decoded stream out to N consumer windows, watches for stalls/dead windows and relaunches them, handles backoff/reconnect and the offline-fallback loop) and `App` (the Tkinter GUI — settings, monitor selection, theme, menu, watchdog tick). `src/monitor_utils.py` wraps `screeninfo` for monitor enumeration/geometry math (pure functions, easy to unit-test). `src/winkiosk.py` holds Windows-only kiosk primitives (DPI awareness, keep-awake, single-instance lock, window-rect enforcement) that are safe no-ops on other platforms. See ARCHITECTURE.md for the full design writeup (threading model, reconnect/self-heal state machine, settings persistence).

**Tech stack:** Python 3.11+, Tkinter, yt-dlp, ffmpeg/ffplay (subprocess), screeninfo, psutil, appdirs, PyInstaller

## Subsystems

### Playback engine (Player) — `src/video-tiler.py`
Owns the entire lifecycle of a video wall: builds the yt-dlp/ffmpeg/ffplay commands, fans one downloaded stream out to N consumer windows (or one ffplay per monitor in multi-monitor mode), and keeps the wall alive — stall detection, per-window relaunch, backoff/reconnect, offline-fallback playback, and window-rect/fullscreen enforcement.

**Key files**
- `src/video-tiler.py` — class `Player` (lines ~446-1298): `_start`/`run` drive the engine; `_spawn_windows`/`_fanout`/`_deliver` handle the single-download-to-N-windows fanout; `_relaunch_consumer`/`_dead_window_indices`/`_stalled` do self-heal; `_play_offline_fallback` handles the offline loop; `_enforce_rect_async` pins exact per-monitor geometry

**Entry points**
- Player.__init__(app, url, divisions)
- Player.run() (worker thread)
- Player.stop()

**Commands**
```
python src/video-tiler.py
```

**Depends on**
- Windows kiosk helpers (src/winkiosk.py)
- Monitor geometry (src/monitor_utils.py)
- external: yt-dlp, ffmpeg, ffplay on PATH

**Gotchas**
- One download is fanned out to many windows for CPU/bandwidth savings — don't naively spawn one yt-dlp per tile.
- Multi-monitor uses one ffplay per screen (a single spanning window was found unreliable) — see README 'Multi-monitor' note.
- Offline handling is a cheap connectivity probe, not reconnect churn — changing the retry loop affects the 'never freeze while offline for hours' guarantee (v1.2/v1.3 hardening).

### GUI / App shell — `src/video-tiler.py`
Tkinter application window: URL/grid-size inputs, monitor picker, theme (light/dark), menu (update yt-dlp, check for updates, help), settings load/save, and the watchdog tick that supervises the active Player.

**Key files**
- `src/video-tiler.py` — class `App(tk.Tk)` (lines ~1299-2030): create_widgets/create_menu build the UI; play_video/stop_video drive a Player; _watchdog_tick supervises it; save_all_settings/load_all_settings persist state; update_yt_dlp/check_for_updates handle self-update

**Entry points**
- main() (module bottom, line ~2031)

**Commands**
```
python src/video-tiler.py
```

**Depends on**
- Playback engine (Player)
- read_settings/write_settings (JSON settings file)

**Gotchas**
- Settings file path and log path come from `appdirs`; see README 'Logs / troubleshooting' for exact per-OS paths.
- `_pump_ui_queue`/`post_ui` marshal calls from the Player worker thread back onto the Tk main thread — never touch widgets directly from `Player.run()`.

### Monitor geometry — `src/monitor_utils.py`
Small, dependency-light wrapper around `screeninfo` for enumerating monitors, picking the primary/selected ones, and computing per-tile crop/scale filters and window placement for a given monitor and grid size.

**Key files**
- `src/monitor_utils.py` — list_monitors, select_monitors, tile_filter_for (ffmpeg filter string per monitor+divisions), window_opts_for, describe

**Entry points**
- list_monitors()

**Depends on**
- screeninfo

**Gotchas**
- Pure functions with no Tk/subprocess dependency by design — keep it that way, it's what makes tests/test_monitor_utils.py fast and dependency-light.

### Windows kiosk hardening — `src/winkiosk.py`
Windows-only primitives for running as an unattended kiosk: Per-Monitor-v2 DPI awareness, keep-the-machine-awake while a show is wanted, a single-instance lock, and a watchdog that re-enforces exact window rects (fighting DPI/monitor-hotplug drift).

**Key files**
- `src/winkiosk.py` — set_dpi_awareness, keep_awake, acquire_single_instance, enforce_window_rect

**Entry points**
- set_dpi_awareness()
- acquire_single_instance()

**Depends on**
- Windows ctypes/user32 (no-ops on macOS/Linux)

**Gotchas**
- Every function must degrade to a safe no-op on non-Windows — this module is imported unconditionally by src/video-tiler.py.

**Entry points**
